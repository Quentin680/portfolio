Portable Web Radio Project

Design and programming of a portable Web Radio based on an ESP32 and a VS1053 audio codec, allowing Internet stations in MP3 or AAC to be played and controlled remotely from a phone.

Project summary for the portfolio

Context & objective

The objective of this SAE was to build a portable Web Radio using an ESP32 and a VS1053 audio codec. The system had to play Internet radio stations in MP3 or AAC through headphones or speakers. The project also had to include several controls: station switching, volume adjustment, tone adjustment and sound spatialisation activation. The final objective was to control the radio from a phone using the MQTT protocol.

Process & methodology

I began by studying the topic in order to identify the expected features. Then, I discovered the hardware used: the ESP32 board, the Music Maker FeatherWing board and the VS1053 codec. I researched the pins required for SPI communication, then configured the Arduino environment with the necessary libraries. After testing a first Web Radio, I gradually added audio settings such as volume, bass, treble and spatialisation. Finally, I improved Wi-Fi management with WiFiManager, added radio station URL management and set up MQTT to control the radio from a phone.

Feedback

What I learned

This SAE allowed me to better understand how a connected embedded system works. I learned to use an ESP32 with Wi-Fi, communicate with a VS1053 audio codec via SPI and manage an audio stream coming from the Internet. I also improved in Arduino programming, especially by modifying an existing program to add audio control functions. Finally, using MQTT helped me understand how to control a connected object from a phone.

What I would do differently

With hindsight, I would organise the code more from the beginning of the project by clearly separating the parts related to Wi-Fi, MQTT, radio stations and audio processing. I would also document the MQTT topics used more clearly to make testing between the ESP32 and the mobile interface easier. Finally, I would carry out more regular tests after each feature addition to identify errors more quickly.
